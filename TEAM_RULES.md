# Правила роботи в команді (4 людини)

Структура гілок: `master`, `dmitry`, `firstTask`, `yaroslav`, `darina`, `stanislav`.

---

## 1. Структура гілок

| Гілка | Призначення |
|-------|-------------|
| `master` | **Стабільна** версія. Сюди потрапляє лише перевірений код. |
| `dmitry`, `yaroslav`, `darina`, `stanislav` | **Персональні** гілки кожного розробника. |
| `firstTask`, `feature/*`, `fix/*` | **Фіча-гілки** для окремих задач (опціонально). |

---

## 2. Робочий процес (Git Flow)

### Початок роботи над задачею

```bash
git checkout master
git pull origin master
git checkout -b <ім'я-гілки>   # напр. yaroslav або feature/auth
```

### Регулярні коміти

```bash
git add .
git commit -m "Короткий опис змін"
```

- Комітити часто, з осмисленими повідомленнями.
- Один коміт = одна логічна зміна.

### Синхронізація з `master`

Перед створенням Pull Request:

```bash
git checkout master
git pull origin master
git checkout <твоя-гілка>
git merge master            # або git rebase master
# вирішити конфлікти, якщо є
git push origin <твоя-гілка>
```

---

## 3. Pull Request (PR)

1. **Створити PR** на GitHub з твоєї гілки в `master`.
2. **Описати**, що зроблено.
3. **Призначити рев'юера** (мінімум 1 інший член команди).
4. **Не мержити самостійно** — чекати approve.

---

## 4. Code Review

- Кожен PR має пройти рев'ю **хоча б однієї** іншої людини.
- Рев'юер перевіряє:
  - Код працює і не ламає існуючий функціонал.
  - Стиль коду відповідає проєкту.
  - Немає очевидних багів.
- Коментарі — конструктивні, без переходу на особистості.

---

## 5. Merge в `master`

- Тільки через **Pull Request**.
- Тільки після **approve** від рев'юера.
- Після merge — **видалити** гілку (GitHub пропонує автоматично).

---

## 6. Комунікація

- **Щоденний статус** (коротко): що зробив, що планую, чи є блокери.
- Використовувати **Issues / Tasks** для трекінгу задач.
- Якщо конфлікт — вирішувати разом, не перезаписувати чужий код без узгодження.

---

## 7. Заборонено

- ❌ `git push --force` в `master`.
- ❌ Комітити напряму в `master` (без PR).
- ❌ Мержити свій PR без рев'ю.
- ❌ Залишати незакомічені зміни надовго.

---

## 8. Рекомендації

- ✅ Пушити свою гілку на GitHub щодня (бекап).
- ✅ Перед початком роботи — `git pull`.
- ✅ Тестувати локально перед створенням PR.
- ✅ Використовувати `.gitignore` для `node_modules`, `.env`, тощо.

---

## Приклад workflow для одного розробника

```bash
# 1. Оновити master
git checkout master && git pull

# 2. Перейти на свою гілку
git checkout darina

# 3. Синхронізувати з master
git merge master

# 4. Працювати, комітити
git add . && git commit -m "Add profile page"

# 5. Запушити
git push origin darina

# 6. Створити PR на GitHub: darina → master

# 7. Після approve — merge через GitHub
```
